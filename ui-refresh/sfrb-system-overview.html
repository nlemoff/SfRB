Why it existsPurpose & philosophy

SfRB is a local-first, CLI-first resume editor. One command initializes a workspace, one command opens a browser editor served by a local bridge, one command exports a PDF — and every one of those paths reads and writes the same single JSON file.

Most resume tools force one of three bad tradeoffs:

  • Document editors are easy to write in but weak on layout control.
  • Design tools give precise layout but become brittle the moment content changes.
  • AI resume tools suggest edits without a trustworthy local source of truth.

SfRB's answer is a different contract, and everything in the system derives from it:

1. One canonical document

resume.sfrb.json is the only authoritative state. Browser, CLI, export, and templates all derive from it. Every mutation passes schema + physics validation before any byte reaches disk.

2. Document and design editing

A workspace declares its physics — document (flowing text) or design (fixed frames) — and the editor offers guided lenses appropriate to that physics over the same model.

3. Chrome-free export

The PDF is rendered from a clean /print presentation surface, never from the editor DOM. What you export carries zero editor affordances.

4. AI proposes, never bypasses

The layout consultant can suggest overflow fixes, but proposals pass the same local validation and require explicit acceptance. Provider keys never leave the local bridge.

5. Explicit failure beats silent degradation

Content that doesn't fit, mutations that are rejected, AI that is unavailable — the product says so visibly. Nothing fails quietly.

6. The one-page contract

v1.0 deliberately models one page. Overflow is measured and gates the export rather than paginating. The consultant exists to help you fit, not to add pages.

Who it's for

Two audiences, one tool: a non-technical user who runs sfrb init / sfrb open and edits in the browser without ever seeing JSON, and a technical user who scripts the same edits through sfrb edit, keeps their resume in git, and trusts that the file on disk is the whole truth. CLI parity is a hard rule: every meaningful editor action must be representable as a structured operation invokable from the CLI.

The pastHow SfRB got here

The project grew through five requirement-driven milestones (R001–R018), survived the loss of an entire editor implementation, and was completed in June 2026 by a greenfield rebuild of that lost work.

  • Milestone 1 · Foundation & physics shipped

    The canonical workspace contract

    Workspace init (R002), the Zod document schema with generated schema.json, document-vs-design physics validation (R005), the local Vite bridge (R003), mode-aware browser editing with persistence (R004), and the first AI consultant loop (R006). Established the rule everything else obeys: one document, one validated write path.

  • Milestone 2 · Resume engine & guided editing original lost rebuilt 2026

    The rich editor that vanished

    The deep editor — three-lens guided editing (R008), structured CLI-parity operations (R009), mode reconciliation (R012), freeform element editing (R014), tile grouping/locking (R015), sleek presentation (R011) — was designed and built in a separate worktree whose code never reached this repository's lineage. Only the design documents survived. For months, main shipped a UI that honestly said "Freeform lens — not shipped yet."

  • Milestone 3 · Export & presentation depth shipped

    PDF export that matches the built resume

    The shared /print surface powering both browser preview and CLI export (R013), deterministic ready/risk/blocked export markers, and Playwright-driven PDF generation. The artifact path is chrome-free; the preview path may expose diagnostics.

  • Milestone 4 · Template & theme system shipped

    Typography as a typed contract

    Three first-party templates (default, classic, modern) selectable from CLI or browser (R018). The binding rule: templates are typography, not geometry — a Theme exposes fonts, colors, and per-block spacing, never page size, margins, or frame boxes. Registry and theme map are aligned at compile time.

  • Milestone 5 · Distribution re-scoped

    The git-clone decision

    Originally an npm-publishing milestone, never started. A deliberate 2026 scope decision re-cast SfRB as a polished git-clone tool: no package publishing, no release automation, no CHANGELOG — docs and verification target the clone-build-run path instead. npm publishing stays a non-goal until a maintainer opts in.

  • June 2026 · PR #19 the one-shot rebuild

    Finishing the app

    A single 10-commit branch (feat/v1-finish-the-app) rebuilt the lost Milestone-2 editor greenfield on today's foundation and finished the product: the 13-operation vocabulary shared by bridge and the new sfrb edit, the tile lens (split/group/lock/canonical group moves), the freeform lens (direct move/resize/dividers), explicit freeform-exit reconciliation, a keyboard-operable accessible shell, a WCAG-AA template contrast gate, and honest docs. The test suite grew from 26 files / 96 tests to 38 files / 172 tests, plus six end-to-end smoke verifiers. CI green.

  • Next · Distribution, automation & ecosystem current focus

    Make it easy to install, verify, extend, maintain

    See The future below.

The presentThe canonical document model

resume.sfrb.json separates semantics (sections and blocks of text) from geometry (pages and frames), connected by IDs. Schema version is 1; every 2026 addition was additive and defaulted, so older documents load unchanged.

{
  "version": 1,
  "metadata": { "title", "locale", "starter", "template": { "id", "version" } },

  "sections": [                       // semantic order
    { "id", "title", "blockIds": ["…"] }
  ],
  "blocks": [
    { "id",
      "kind": "heading | paragraph | bullet | fact | divider",
      "text",                         // required unless kind === 'divider'
      "splitFrom"?: "originalBlockId" // provenance from split-block
    }
  ],

  "layout": {
    "pages":  [ { "id", … } ],        // one page (v1.0 contract)
    "frames": [
      { "id", "blockId", "pageId",
        "box": { "x", "y", "width", "height" },   // points
        "zIndex",
        "placement": "managed | free" }           // freeform reconciliation state
    ],
    "frameGroups": [
      { "id", "pageId", "frameIds": [ ≥2 ], "locked": boolean }
    ]
  }
}

Schema invariants (enforced by Zod superRefine)

  • All IDs unique within their collection; frames reference existing blocks/pages.
  • Non-divider blocks require non-empty text ("Block text is required"); dividers carry text: "".
  • Frame-group invariants: unique group IDs, every member frame exists, a frame belongs to at most one group, member pages match the group's page.
  • Physics overlay: document workspaces forbid frames and frame groups entirely; design workspaces require every block to be frame-backed.

The machine-readable contract is generated to schema.json (npm run schema:generate) and CI asserts it never drifts from the Zod source (npm run schema:check).

The presentThe 13-operation mutation vocabulary

Every editor action is a structured operation — a Zod discriminatedUnion on op — applied by a pure function and validated before any write. The browser's {operation} route and sfrb edit run the same code, so the two front doors cannot fork the mutation model.

set-section-titleRenames a section.Section must exist.
set-templateApplies a registered template's metadata.Template ID must be in the registry.
set-block-textReplaces a block's text.Block must exist; text non-empty.
insert-blockAdds a block (optionally with a frame, e.g. freeform dividers).ID collisions rejected; design physics requires the frame.
remove-blockRemoves a block with a full cascade.Cascade: section membership → drop empty section → drop frame → group membership → dissolve groups under 2 members. Removing the last block is rejected.
split-blockAtomically replaces one block with N derived segments carrying splitFrom provenance.Frame-backed originals require matching frame boxes for each segment; frameless originals forbid them.
set-frame-boxMoves and/or resizes one frame (drags, nudges, consultant accepts).Rejected for locked-group members ("move the group or unlock it") and for free-placed frames unless flagged asFreeform (the freeform lens's acknowledgment).
move-groupTranslates every member of a locked group by (dx, dy) as one canonical write.Group must exist and be locked.
group-framesCreates a frame group from ≥2 frames.Members must exist, share one page, be ungrouped, and not be freeform-placed.

CLI parity in practice

# discover the vocabulary
node dist/cli.js edit --list-ops

# inline op, machine-readable envelope
node dist/cli.js edit --cwd ~/my-resume --json \
  --op '{"op":"set-block-text","blockId":"summaryBlock","text":"New summary."}'

# from a file, or from stdin
node dist/cli.js edit --cwd ~/my-resume --op-file ./operation.json
echo '{"op":"set-title","title":"My Resume"}' | node dist/cli.js edit --cwd ~/my-resume

The --json envelope reports {ok, code, op, writeOutcome: "written" | "no_write", issues, documentPath, configPath}; invalid operations exit non-zero without writing. If the browser editor is open, it picks CLI edits up live through bootstrap reconciliation.

The presentThe bridge HTTP contract

Four load-bearing routes. Bridge update events are invalidation signals, not authoritative payloads — the browser always reconciles full state from bootstrap.

RoutePurposeFailure codes
GET /__sfrb/bootstrapFull-state reconciliation source: workspace paths, physics, starter, AI status, and the entire canonical document.—
POST /__sfrb/editorAll browser mutations, as exactly one of {document} (full document, legacy) or {operation} (structured op). Validation runs before any write.400 request_invalid (both/neither key), 400 operation_invalid (path-aware issues, no write), 422 document_invalid, 409 physics_invalid, 500 persistence_failed
POST /__sfrb/consultantAI overflow proposals. Browser sees only validated proposal payloads or sanitized failures.502 provider_auth / proposal_rejected, 503 provider_rate_limited / provider_unavailable, configuration_missing
GET /printShared presentation surface. ?mode=artifact is chrome-free (the export path); preview mode may carry diagnostics, plus data-template-id/-version markers.—

The presentPhysics & the three lenses

A workspace declares its physics at init; the editor then offers lenses — guided views over the one canonical document. Lenses are browser-local state; the document never records which lens you're in.

Lensdocument physicsdesign physicsWhat it's for
Text✅ (default)✅Flow-ordered inline text editing. Click a block, type, save is debounced and emitted as set-block-text.
Tile— (needs frames)✅ (default)Structured layout: drag managed frames, split multi-line tiles into provenance-carrying segments, group/ungroup, lock compositions that then move only as one canonical move-group.
Freeform— (needs frames)✅Direct manipulation: grab any element anywhere, resize from the SE corner handle, add/remove divider elements. Exits through explicit reconciliation.

Unavailable lenses say why ("Requires a design workspace — physics is set at init") instead of disappearing. The switcher is a keyboard-operable radiogroup with roving tabindex.

The presentFreeform reconciliation — mode changes without surprise

The hardest design problem in a hybrid document/design editor is what happens when freeform placement meets managed layout. SfRB's answer: an explicit, placement-only state machine.

  • Geometry continuity is sacred. Freeform commits persist immediately (debounced set-frame-box with asFreeform); reconciliation later changes only placement. Users never watch their layout snap back.
  • Kept-free frames stay honest. A free frame in the tile lens shows blocked diagnostics on any drag, nudge, or grouping attempt — the op layer rejects it and the UI says why ("edit it in the freeform lens or reconcile with rejoin"). Text editing of its block stays allowed.
  • Overflow context carries across the switch — the transition strip echoes any live overflow diagnosis so a lens change never hides a problem.

The presentThe AI layout consultant

The consultant is deliberately the only AI surface in v1.0: it proposes overflow fixes — never content. BYOK (bring your own key), optional, and fully subordinate to local validation.

  • Flow: editor detects overflow → user requests a proposal → bridge calls the configured provider (DeepSeek default at deepseek-v4-flash; OpenAI and Anthropic supported) → response is parsed, schema-validated, and bounds-checked against page geometry (off-page boxes are rejected as proposal_rejected) → browser shows a ghost preview → explicit accept dispatches an ordinary set-frame-box through the same write path as a manual drag.
  • Failure classification (added 2026): HTTP 401/403 → provider_auth (bridge 502), 429 → provider_rate_limited (503), other failures → provider_unavailable. Messages are status-shaped only — raw provider response bodies never reach the browser or logs.
  • Skipped is a first-class state: init --skip-ai workspaces show an honest "AI setup is skipped" panel; text/tile/freeform editing is fully functional without any key.

The presentAccessibility & presentation quality

Keyboard-operable canvas

Frames are tabbable with descriptive labels; Enter opens text editing, Escape exits/clears, arrows nudge 1pt and Shift+arrows 10pt through the same commit path as drags. Blocked targets show a diagnostic instead of silently not moving.

Semantic shell

Landmarks (main, header, labeled asides), lens switcher as a radiogroup with roving tabindex, save state on aria-live="polite", blocked-action notes as role="status", dialog focus-trapped with focus restoration.

Automated contrast gate

Every registered template theme must pass WCAG AA (4.5:1) text-vs-page contrast, computed by web/src/presentation/contrast.ts and enforced by a test — a fourth template that fails contrast cannot ship.

Calm presentation

The canvas is primary; diagnostics (bridge status, AI panel, payload preview, workspace paths) live in a collapsed inspector. A light, low-chrome shell with one injected stylesheet replaced the old inline-style strings.

The presentSecurity model

RuleMechanism
No secrets on disk or in the browsersfrb.config.json stores only provider metadata and the name of an env var (e.g. DEEPSEEK_API_KEY); the bridge reads the key from the environment at call time. Workspace .gitignore protects the config file anyway.
Browser never sees provider internalsConsultant responses are validated bridge-side; the browser receives only schema-valid proposal payloads or sanitized, code-classified failures. No raw bodies, no auth headers.
One validated write pathAll disk I/O is owned by the bridge / operation runner. Parse → apply → schema validate → physics validate → write; any failure means no write. There is no second path to corrupt the document.
Input validation at every boundaryZod schemas validate CLI arguments, file content, bridge request bodies, and provider responses; untrusted input is unknown until narrowed.
Local-first by constructionThe bridge binds locally; the resume never leaves the machine except as the PDF you export or the provider prompt you explicitly configure.

The presentTesting & verification infrastructure

38 vitest files / 172 tests plus six end-to-end smoke verifiers, all green in CI (build-test is a required check on main).

  • Layered suites: document/schema unit tests, operation contract tests, bridge HTTP contract tests, CLI tests against the built dist/cli.js, presentation tests (including the byte-stability suite for the default template and the contrast gate), and real-Chromium browser tests (tests/web/) driving the actual bridge.
  • Build discipline: a vitest global setup builds dist/ exactly once before workers start; tests never rebuild mid-suite (this eliminated a whole class of rebuild-race flakes), and parallelism is capped at four workers.
  • Smoke verifiers (npm run verify:smoke): editor, consultant, CLI edit, tile engine, freeform, and reconciliation — each drives the built CLI and a real browser through a user journey. A package-install smoke (npm run verify:package) packs the tarball and exercises the installed CLI.
  • The frozen selector contract: every id/data-testid referenced by tests and verifiers is frozen — elements may move freely, but renaming or dropping a selector is a breaking change. This is what made the 2026 shell rewrite provable: the pre-existing editor and consultant smokes had to pass unmodified.
  • Schema drift gate: schema.json is generated from the Zod source and CI fails if it drifts.

Honesty sectionKnown limitations & honest edges

  • One page, period. Long resumes must be trimmed to fit (the consultant helps); there is no second page and no pagination setting. This is a contract, not a bug.
  • Consultant vs. locked/free frames: accepting an AI proposal that targets a locked-group member or a freeform-kept frame fails visibly as operation_invalid in the consultant panel — the documented honest edge of "AI passes the same rules as you."
  • Canvas serif is pinned to Times New Roman because overflow-measurement expectations across tests and smokes are tuned to those metrics. Changing the canvas font means retuning the measurement suite.
  • Design physics is set at init. There is no in-place document↔design physics migration; tile and freeform lenses simply explain their unavailability in document workspaces.
  • The legacy {document} route remains alongside {operation} for back-compat. New features must use operations; whole-document writes are not the future of the mutation model.
  • Smokes run locally, not in CI. CI runs build + schema check + the full vitest suite; the six browser smokes are a pre-merge local gate (moving them into CI is current-focus work).
  • Distribution friction is real: clone → npm install → npm run build → node dist/cli.js is the supported path. That friction is a chosen tradeoff, revisitable per the deferred-decisions table.
